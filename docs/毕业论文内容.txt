摘  要

随着云计算和大数据技术的快速发展，文件共享已成为现代工作与生活中的基本需求。然而，传统文件共享系统普遍依赖于对服务端的信任，一旦服务器遭受攻击，用户数据的机密性将完全丧失。为解决这一问题，本文设计并实现了一个基于椭圆曲线密码（ECC）的端到端加密文件共享系统。系统采用混合加密架构：文件内容使用AES-256-GCM对称加密算法保护机密性和完整性，为每个文件生成独立的256位AES密钥；AES密钥则通过ECIES（椭圆曲线集成加密方案）使用授权用户的ECC公钥进行封装传输。系统的ECC密钥对由浏览器端Web Crypto API生成，选用P-384曲线以确保安全强度。私钥经用户口令和PBKDF2算法加密后存储在浏览器本地，服务器仅持有公钥，实现了"零知识"安全模型。多用户文件共享采用密钥清单机制，文件加密一次，通过为每个授权用户单独加密AES密钥实现安全共享，避免了文件重复加密的开销。系统后端基于Python FastAPI框架开发RESTful API，使用SQLAlchemy ORM管理SQLite数据库；前端采用原生HTML5/CSS3/JavaScript实现单页应用，通过Web Crypto API在浏览器端完成全部密码学操作。系统实现了用户注册登录、文件加密上传与下载、多用户文件分享、批量操作、操作审计日志、频率限制等完整功能。经测试验证，系统能够有效保护文件在存储和传输过程中的机密性，在多用户共享场景下表现出良好的安全性与可用性。

关键词：椭圆曲线密码；端到端加密；文件共享；ECIES；混合加密；密钥管理

ABSTRACT

With the rapid development of cloud computing and big data technologies, file sharing has become a fundamental requirement in modern work and daily life. However, traditional file sharing systems generally rely on trust in the server side. Once the server is compromised, the confidentiality of user data is completely lost. To address this issue, this thesis designs and implements an end-to-end encrypted file sharing system based on Elliptic Curve Cryptography (ECC). The system adopts a hybrid encryption architecture: file content is protected by AES-256-GCM symmetric encryption for confidentiality and integrity, with a unique 256-bit AES key generated for each file. The AES keys are encapsulated and transmitted using the ECIES (Elliptic Curve Integrated Encryption Scheme) with the authorized recipient ECC public key. ECC key pairs are generated using the Web Crypto API in the browser, with the P-384 curve selected for sufficient security strength. Private keys are encrypted with the user password via PBKDF2 and stored locally in the browser, while the server only holds the public key, achieving a zero-knowledge security model. Multi-user file sharing adopts a key inventory mechanism, where the file is encrypted only once, and the AES key is individually encrypted for each authorized user, avoiding redundant file encryption. The backend is developed using the Python FastAPI framework to provide RESTful APIs, with SQLAlchemy ORM managing an SQLite database. The frontend is a single-page application built with native HTML5, CSS3, and JavaScript, performing all cryptographic operations in the browser through the Web Crypto API. The system implements comprehensive features including user registration and login, encrypted file upload and download, multi-user file sharing, batch operations, audit logging, and rate limiting. Testing has demonstrated that the system effectively protects the confidentiality of files during storage and transmission, exhibiting strong security and usability in multi-user sharing scenarios.

Keywords: Elliptic Curve Cryptography; End-to-End Encryption; File Sharing; ECIES; Hybrid Encryption; Key Management

第1章 绪论
1.1 研究背景与意义
随着信息技术的快速发展和互联网的广泛普及，文件共享已成为现代社会工作协作与信息交流的核心方式。无论是企业内部文档的协同编辑，还是个人之间的照片、视频分享，高效便捷的文件共享服务都是不可或缺的数字基础设施。根据Statista的统计，全球云存储市场规模已从2020年的700亿美元增长至2025年的超过1700亿美元，文件共享和同步服务作为其中的核心组成部分，其重要性日益凸显。
然而，在文件共享带来便利的同时，数据安全与隐私保护问题也日益严峻。2024年IBM发布的《数据泄露成本报告》显示，全球数据泄露的平均成本已达到488万美元。近年来，Dropbox、OneDrive等主流云存储服务均发生过不同程度的安全事件，导致大量用户文件的泄露。传统文件共享系统的安全模型主要依赖于对服务端的信任——服务端管理着加密密钥和访问控制，用户数据在服务端以明文或服务端可解密的形式存储。这种架构存在根本性的安全缺陷：一旦服务器被攻破，或者服务提供商本身存在内部威胁，所有用户的数据将完全暴露。
端到端加密（End-to-End Encryption, E2EE）技术为解决这一问题提供了一条有效途径。在E2EE架构中，数据在发送端加密后传输，只有预期的接收端才能解密，中间节点（包括服务端）无法访问明文内容。这一技术已在即时通讯领域获得广泛应用（如Signal、WhatsApp），但在文件共享系统的应用仍处于发展阶段。实现E2EE文件共享的核心挑战在于：如何在保证加密强度的同时，实现灵活的多用户共享和高效的密钥管理。
椭圆曲线密码学（Elliptic Curve Cryptography, ECC）是公钥密码学的重要分支，由Miller和Koblitz于1985年独立提出。ECC的安全性基于椭圆曲线离散对数难题（ECDLP），目前尚未发现能在多项式时间内求解ECDLP的经典算法。与传统的RSA算法相比，ECC在相同安全强度下所需的密钥长度显著更短。例如，256位ECC密钥的安全强度约等于3072位RSA密钥，而384位ECC密钥的安全强度已达192位，满足绝大多数应用场景的长期安全需求。更短的密钥意味着更小的存储开销、更快的运算速度和更低的带宽消耗，这使得ECC特别适合于移动设备、物联网设备和Web应用等资源受限的环境。随着Web Crypto API标准的成熟和主流浏览器的广泛支持，在浏览器端直接进行ECC密钥生成和加密运算已成为现实，为构建纯客户端加密的Web应用提供了坚实的技术基础。
基于以上背景，本课题设计并实现了一个基于ECC的认证文件共享系统。该系统采用混合加密架构，文件内容使用AES-256-GCM对称加密保护，AES密钥通过ECIES方案使用授权用户的ECC公钥进行安全分发。所有密码学操作均在用户浏览器端完成，服务端仅存储密文数据和公钥，实现了"零知识"安全模型。系统提供了完整的Web应用界面，支持用户注册登录、文件加密上传与下载、多用户文件分享、批量操作等功能，能够在保障数据机密性的同时，提供良好的用户体验。
1.2 国内外研究现状
在文件共享安全领域，国内外学者和工业界已开展了广泛的研究与实践。根据技术路线的不同，现有方案可大致分为以下几类：
（1）基于公钥基础设施（PKI）的方案。传统的安全文件共享系统通常依赖于PKI体系，服务端持有用户的加密密钥或能够解密用户数据。Tresorit和TeamDrive是这一领域的代表性商业产品，采用混合加密架构：文件用AES-256对称加密，密钥用RSA或ECC非对称加密分发给授权用户，但客户端实现通常是闭源的，用户无法验证其安全性。
（2）基于属性基加密（ABE）的方案。张玉清等人对基于属性加密的云存储数据访问控制方案进行了系统综述[1]，指出ABE能够实现一对多的细粒度访问控制，但存在密文长度随属性数量线性增长的问题，计算复杂度较高，在移动端等资源受限环境中的实用性受限。
（3）基于代理重加密的方案。张晓均等人提出了支持隐私保护的可验证云端数据分享方案[2]，利用代理重加密技术实现密文的访问权限转移，无需解密原始文件即可将加密文件的访问权转授给其他用户。但代理重加密的计算复杂度较高，且需要可信的代理服务器。
（4）基于区块链的去中心化方案。林洁和等人设计了基于ECC的区块链数据共享系统[3]，利用智能合约管理访问权限，提升了系统的透明性和抗篡改能力。陈志翔等人提出了基于区块链和椭圆曲线的改进多权威属性访问控制方案[4]。李振宇等人对基于ECC的区块链数据共享系统的安全认证与激励机制进行了研究[5]。这些方案利用区块链的去中心化特性增强了系统的可信性，但在性能和用户体验方面仍有待优化。
（5）ECC算法与协议研究。汪朝晖对SM2椭圆曲线公钥密码算法进行了全面综述，介绍了我国自主知识产权的ECC标准算法[6]。武传坤分析了物联网安全的关键技术与挑战，指出ECC在资源受限设备中的天然优势[7]。在身份认证方面，邓淼磊等人对元宇宙环境下的身份认证技术进行了系统综述[8]。在密钥管理层面，张艳丽等人提出了基于椭圆曲线的可验证多秘密共享方案[9]，为多用户密钥分发提供了理论基础。孙晓玲等人提出了改进的高效动态可搜索加密方案[10]，为云端文件的安全检索带来了新思路。
综上所述，现有研究在安全文件共享领域取得了丰富的成果，但仍存在一些不足：商业方案多为闭源实现，学术方案往往侧重于特定领域的优化而非通用系统的构建。本课题在现有研究基础上，设计并实现了一个基于ECC的、全客户端加密的、开源的Web文件共享系统，旨在提供一种兼顾安全性、易用性和可部署性的实用方案。
1.3 主要研究内容
本课题的主要研究内容包括以下几个方面：
（1）ECC密码算法模块的设计与实现。基于Web Crypto API实现ECC密钥对的生成，选用P-384曲线以保证足够的安全强度。设计并实现基于ECIES的密钥封装方案，通过临时ECC密钥对、ECDH密钥协商和HKDF密钥派生，安全地将AES会话密钥从文件所有者传递给各授权用户。研究合理的密钥管理生命周期，包括密钥的生成、存储、分发和销毁。
（2）混合加密文件存储架构的设计。文件内容采用AES-256-GCM认证加密模式，同时保证机密性和完整性，每个文件使用独立的随机生成AES密钥。设计"密钥清单"（FileKey表）数据结构，维护每个文件与各授权用户加密AES密钥的映射关系，实现"文件加密一次，多用户安全共享"的目标。文件在客户端加密后上传，服务端仅存储密文文件，无法解密文件内容。
（3）用户认证与密钥管理方案的设计。用户注册时在浏览器端生成ECC密钥对，公钥上传至服务端存储，私钥经用户口令使用PBKDF2算法加密后保存在浏览器localStorage中，实现客户端密钥托管。基于JWT实现无状态的用户身份认证，防止会话伪造与劫持。设计安全的密码修改流程，确保私钥在密码变更时能够无缝迁移。支持私钥的导出与恢复功能，防止因浏览器缓存清理导致的密钥永久丢失。
（4）Web前后端系统的完整实现。后端采用Python FastAPI框架开发RESTful API，使用SQLAlchemy ORM操作SQLite数据库，实现用户管理、文件元数据管理、加密密钥存储、操作审计日志等核心业务逻辑。前端采用原生HTML5/CSS3/JavaScript实现单页应用，利用Web Crypto API在浏览器端完成全部的ECC密钥生成、ECIES加解密、AES-GCM加解密等密码学操作，形成"零知识"安全架构——服务端对文件明文完全无感知。系统支持文件上传/下载/分享/删除/重命名/批量操作等完整功能。配置速率限制机制保护敏感API端点，SQLite启用WAL模式提升并发性能。

2.1 椭圆曲线密码学
椭圆曲线密码学（Elliptic Curve Cryptography, ECC）是一种基于椭圆曲线数学理论的公钥密码体制。与传统的基于大整数分解难题的RSA算法不同，ECC的安全性建立在椭圆曲线离散对数难题（Elliptic Curve Discrete Logarithm Problem, ECDLP）之上。
椭圆曲线是由Weierstrass方程 y^2 = x^3 + ax + b 定义的平面曲线，其中判别式 Δ = -16 (4a^3 + 27b^2) ≠ 0 以保证曲线是非奇异的。在密码学应用中，曲线的点和坐标定义在有限域GF(p)上（p为大素数）。椭圆曲线上的点集和无穷远点O构成一个Abel群，群运算是点加法。椭圆曲线离散对数难题可描述为：给定椭圆曲线E上的基点G和点Q = kG（k为标量乘），在已知G和Q的情况下，求出k的值在计算上是不可行的。
与RSA算法相比，ECC在相同安全强度下所需的密钥长度显著更短，具体对比如表2.1所示。256位ECC密钥提供的安全强度约等于3072位RSA密钥，而384位ECC密钥的安全强度可达192位。这意味着ECC在存储、传输和计算效率方面具有显著优势，更适合资源受限的场景，如移动设备、物联网和Web应用。
本系统选用P-384（secp384r1）椭圆曲线。P-384是由NIST（美国国家标准与技术研究院）推荐的384位素域椭圆曲线，其安全强度为192位，足以满足绝大多数商业应用的长期安全需求。该曲线已被ANSI X9.62、IEEE P1363和NIST SP 800-56A等标准采纳。该曲线的域参数为：p = 2^384 - 2^128 - 2^96 + 2^32 - 1，这是一个384位的素数，确保了有限域GF(p)上的离散对数问题的计算困难性。
椭圆曲线Diffie-Hellman（ECDH）密钥协商是ECC在密钥交换方面的核心应用。假设通信双方Alice和Bob分别拥有密钥对（d_A, Q_A = d_A·G）和（d_B, Q_B = d_B·G），则双方可以通过交换公钥计算出共享秘密S = d_A · Q_B = d_B · Q_A = d_A · d_B · G。由于ECDLP的困难性，窃听者无法从公开信道中传输的Q_A和Q_B推导出共享秘密S。本系统中的ECIES加密方案即基于ECDH密钥协商实现AES密钥的安全传递。
2.2 AES-GCM对称加密
高级加密标准（Advanced Encryption Standard, AES）是目前应用最广泛的对称分组密码算法。AES支持128位、192位和256位三种密钥长度，分别对应10轮、12轮和14轮的加密变换。本系统选用AES-256，即256位密钥，以提供最高的安全强度。
GCM（Galois/Counter Mode）是一种认证加密模式，它在CTR（Counter）加密模式的基础上，加入了GMAC（Galois Message Authentication Code）消息认证码的计算。GCM模式同时提供数据机密性和完整性保护，能够检测到密文在传输或存储过程中是否被篡改。GCM模式的加密过程如下：使用AES分组密码以CTR模式加密明文，同时利用GHASH函数计算认证标签。解密时先验证认证标签的有效性，只有在标签验证通过后才输出解密结果，否则拒绝解密。这种设计有效防止了选择密文攻击和密文篡改攻击。
在本系统中，每个文件在上传前随机生成一个独立的256位AES密钥，使用AES-256-GCM模式加密文件内容。加密时随机生成12字节的初始向量（IV），与密文拼接后上传。解密时从密文中提取IV后使用AES-GCM解密，浏览器会自动验证认证标签，若验证失败则说明文件在传输或存储过程中被篡改，解密操作将抛出异常。这种设计确保了文件在不可信信道和不可信存储介质上的机密性和完整性。
2.3 ECIES混合加密方案
椭圆曲线集成加密方案（Elliptic Curve Integrated Encryption Scheme, ECIES）是一种混合加密方案，它将公钥加密和对称加密的优点结合起来。ECIES的工作流程为：发送方生成一个临时ECC密钥对；利用ECDH密钥协商，使用自己的临时私钥和接收方的公钥计算出共享秘密；将共享秘密通过HKDF密钥派生函数转换为AES对称密钥；使用AES-GCM加密明文；将临时公钥、IV和密文打包发送。接收方使用临时公钥和自己的私钥执行ECDH得到相同的共享秘密，经HKDF派生后解密。
在本系统中，ECIES方案用于安全封装文件的AES密钥。当用户上传文件时：（1）生成临时256位AES会话密钥，用于加密文件内容；（2）生成临时ECC密钥对（P-384），使用自己的ECC公钥和临时私钥通过ECDH和HKDF派生出密钥加密密钥（KEK）；（3）使用KEK加密AES会话密钥，将临时公钥、IV和加密后的AES密钥拼接为ECIES密文提交给服务器。当用户分享文件时，分享者从服务器获取自己对应的ECIES密文，用自己的私钥解密得到AES会话密钥，然后获取目标用户的ECC公钥，使用新的临时密钥对执行ECIES加密，将新的ECIES密文存入密钥清单，完成安全的分发过程。整个过程确保了AES会话密钥仅在授权用户的浏览器端以明文形式存在，服务器和网络中的任何中间节点均无法访问。
2.4 Web Crypto API
Web Crypto API是W3C制定的Web标准（Web Cryptography API），为浏览器环境提供了原生的密码学操作接口。该API由浏览器原生实现，利用操作系统底层的密码学库（如Windows CNG、macOS Security Framework），具有高性能和高安全性的特点。Web Crypto API提供了SubtleCrypto接口，支持以下关键密码学操作：密钥生成（generateKey）、密钥导入导出（importKey / exportKey）、加密解密（encrypt / decrypt）、签名验证（sign / verify）、密钥派生（deriveKey / deriveBits）、摘要计算（digest）。
本系统充分利用Web Crypto API实现了浏览器端的全部密码学操作，具体包括：使用ECDH算法生成P-384密钥对（generateKey, {name: "ECDH", namedCurve: "P-384"}）；使用AES-GCM进行文件内容的对称加解密（encrypt/decrypt, {name: "AES-GCM", iv: randomIV}）；使用ECDH deriveBits实现密钥协商，结合HKDF deriveKey实现密钥派生，构建完整的ECIES密钥封装方案；使用PBKDF2 deriveKey从用户口令派生加密密钥，保护本地存储的ECC私钥（iterations: 600000, hash: "SHA-256"）。通过Web Crypto API，本系统实现了所有密钥在浏览器端生成、所有加解密运算在浏览器端完成的"零知识"架构，私钥永不以明文形式离开浏览器环境。
2.5 FastAPI与SQLAlchemy
FastAPI是一个用于构建API的现代Python Web框架，基于Starlette和Pydantic。它具有以下突出特点：（1）极高的性能，可与Node.js和Go相媲美；(2）易于使用，开发效率高；（3）自动生成交互式API文档（Swagger UI和ReDoc）；（4）基于Python类型提示的数据验证和序列化。本系统选择FastAPI作为后端框架，利用其异步支持和自动文档生成能力快速构建RESTful API。
SQLAlchemy是Python生态中最流行的ORM框架，提供了完整的SQL工具包和对象关系映射功能。本系统使用SQLAlchemy 2.0版本的DeclarativeBase声明式映射和sessionmaker会话管理，操作SQLite数据库存储用户信息、文件元数据和加密密钥。SQLite选择WAL（Write-Ahead Logging）模式，允许读写操作并发执行，显著提升多用户场景下的数据库性能。同时设置synchronous=NORMAL，在保证数据安全的前提下提高写入性能。
除上述核心技术外，系统还采用了以下辅助技术：python-jose库实现JWT Token的生成与验证，用于无状态的用户身份认证；passlib库实现pbkdf2_sha256密码哈希算法，保护用户登录凭证；Pydantic v2实现请求数据的自动校验和序列化；python-multipart支持文件上传的multipart/form-data解析。前后端通信遵循RESTful架构风格，前端通过fetch API向后端发送HTTP请求，认证信息通过Bearer Token在请求头中传递。

第3章 系统设计
3.1 系统总体架构
本系统采用前后端分离的B/S架构，总体分为三层：表现层、业务逻辑层和数据持久层。表现层为运行在用户浏览器中的单页Web应用（SPA），负责用户交互界面和所有密码学运算；业务逻辑层为FastAPI服务器，负责处理HTTP请求、执行业务逻辑和权限验证；数据持久层为SQLite数据库和本地文件系统，负责存储元数据和加密文件。
系统的核心安全设计原则是"零知识"（Zero-Knowledge）：服务端在任何时刻都无法访问文件明文或加密密钥的明文形式。这一原则通过以下机制实现：（1）ECC密钥对在用户浏览器端生成，私钥经用户口令加密后存储在浏览器localStorage中，服务端仅保存公钥；（2）文件在浏览器端使用随机生成的AES-256密钥加密后上传，服务端仅存储密文文件；（3）AES密钥通过ECIES方案使用目标用户的ECC公钥加密后存储在服务端，服务端无法解密。系统架构如图3.1所示（注：建议在Word中插入Visio或Draw.io绘制的系统架构图）。
系统的数据流向可以概括为三条主要路径：上传路径——用户选择文件，浏览器生成AES密钥加密文件，用自己的ECC公钥加密AES密钥，加密文件和加密密钥一同上传至服务端；下载路径——服务端返回加密文件内容和加密的AES密钥，浏览器使用用户ECC私钥解密AES密钥，再用AES密钥解密文件内容；分享路径——分享者解密自己持有的AES密钥，获取目标用户公钥并重新加密AES密钥，将新的加密密钥存入服务端密钥清单。
3.2 功能模块设计
系统按功能划分为以下五个核心模块：
（1）ECC密码算法模块。负责ECC密钥对的生成、ECIES加密方案的实现。使用Web Crypto API的ECDH算法在浏览器端生成P-384密钥对。ECIES加密过程包括：生成临时ECC密钥对、执行ECDH密钥协商获取共享秘密、通过HKDF将共享秘密派生为AES-GCM加密密钥、加密明文数据。该模块是所有加密操作的基础，封装在JavaScript函数genECC()、eccEncrypt()和eccDecrypt()中。
（2）用户认证与密钥管理模块。负责用户注册、登录、身份认证以及密钥的生命周期管理。注册时用户提供用户名和密码，浏览器自动生成ECC密钥对。公钥提交至服务端的/auth/register端点存储，私钥经PBKDF2从用户口令派生加密密钥后加密，存入浏览器localStorage（key为sk_<username>）。登录时从localStorage获取加密私钥，使用用户输入的口令解密并加载到内存中。认证采用JWT Bearer Token机制，Token由服务端在登录成功后签发，有效期为24小时。前端在每次API请求的Authorization头中携带Token，后端通过get_current_user依赖函数验证Token有效性并解析用户身份。密码修改时，系统先用旧口令解密localStorage中的加密私钥，再用新口令重新加密后存储，确保私钥持续可用。支持私钥导出（提供口令后下载为PEM文件）和导入（从PEM文件恢复到localStorage）功能。
（3）混合加密文件存储模块。负责文件加密、上传、存储和下载的业务流程。上传流程：用户选择文件 → 生成随机AES-256密钥 → AES-GCM加密文件内容（随机IV） → 用用户ECC公钥经ECIES加密AES密钥 → 将加密文件和ECIES密钥密文通过multipart/form-data提交至/files/upload接口。上传过程中边写边校验文件总大小，超过50MB限制时终止上传并删除已写入的临时文件。存储方案：加密文件以UUID命名存储在服务器的uploads/目录下，文件名、所有者和文件大小等元数据存储在files表中。下载流程：服务端通过/files/{id}/download接口以StreamingResponse流式返回加密文件内容，加密密钥通过X-Encrypted-Key响应头以Base64编码传递，前端接收后执行ECIES解密和AES-GCM解密，还原为原始文件并触发浏览器下载。
（4）文件共享与批量操作模块。负责多用户文件分享以及批量文件管理功能。文件分享的核心机制是"密钥清单"：用户A上传文件时在file_keys表中创建一条记录，存储用户A的ECIES加密AES密钥。当用户A要将文件分享给用户B时，先通过/files/{id}/key接口获取自己的ECIES密文，解密得到AES密钥明文；然后通过/users/{username}/public_key接口获取用户B的公钥；最后用用户B的公钥执行ECIES加密，调用/files/{id}/share接口将新的ECIES密文存入file_keys表。批量分享对选中的多个文件逐一执行上述分享流程。批量删除支持用户勾选多个文件后一次性删除，后端验证所有文件的所有权后，删除文件系统中的加密文件和数据库中的元数据及密钥记录。文件重命名仅修改files表中的filename字段，不影响加密文件的实际存储路径。
（5）审计日志与安全防护模块。负责记录用户操作行为和系统安全防护。日志模块记录用户的关键操作，包括登录、上传、下载、分享、删除、重命名、批量删除和修改密码等行为，每条日志包含操作用户ID、操作类型、操作目标、来源IP地址和时间戳，存储在logs表中。安全防护包括：密码长度不低于6位的注册校验；基于滑动窗口的API速率限制，注册接口每IP每分钟限5次、登录接口限10次、修改密码接口限5次，有效防御暴力破解攻击；CORS跨域保护通过FastAPI中间件配置可信来源域名；上传文件大小限制为50MB，防止存储资源滥用。
3.3 数据库设计
系统使用SQLite作为数据库引擎，设计了4张核心数据表，分别为用户表（users）、文件表（files）、文件密钥表（file_keys）和操作日志表（logs）。各表的详细设计如下：
用户表（users）存储用户的基本信息和ECC公钥。字段包括：id（主键，自增整数）、username（用户名，唯一索引，不可为空）、password_hash（密码哈希值，不可为空）、public_key（ECC公钥，以DER二进制格式存储为LargeBinary）、created_at（创建时间）。该表与files表、file_keys表和logs表通过外键建立一对多关联，并设置了级联删除，当用户账户注销时自动清理其拥有的文件、密钥记录和日志。
文件表（files）存储加密文件的元数据。字段包括：id（主键）、filename（用户视角的文件名）、owner_id（外键→users.id，文件所有者）、encrypted_path（加密文件在服务器上的存储路径）、file_size（原始文件大小，字节数）、created_at（创建时间）。该表在owner_id上建立了复合索引以加速按所有者查询。
文件密钥表（file_keys）是实现多用户共享的核心数据结构，存储每个文件对每个授权用户的加密AES密钥。字段包括：id（主键）、file_id（外键→files.id）、user_id（外键→users.id）、encrypted_key（经ECIES加密的AES密钥，以二进制格式存储）。该表在file_id和user_id字段上分别建立了索引。当且仅当file_keys表中存在(file_id, user_id)对时，该用户才有权解密对应文件。文件所有者通过将自己的解密AES密钥用目标用户的ECC公钥重新加密并插入一条新的file_keys记录来完成分享操作。
操作日志表（logs）记录用户的关键操作行为。字段包括：id（主键）、user_id（外键→users.id）、action（操作类型，如login/upload/download/share/delete等）、target（操作目标描述，如文件名或目标用户名）、ip_address（请求来源IP地址）、created_at（操作时间）。该表在user_id上建立了索引以加速按用户的查询。
数据库表之间的关系为：User 1:N File（一个用户拥有多个文件），User 1:N FileKey（一个用户持有多个密钥记录），File 1:N FileKey（一个文件可被多个用户访问），User 1:N Log（一个用户有多条操作日志）。所有外键关系均设置了SQLAlchemy ORM层的relationship，支持级联删除，保证数据一致性。数据库使用SQLite WAL模式（PRAGMA journal_mode=WAL）和NORMAL同步级别（PRAGMA synchronous=NORMAL），在读多写少的文件共享场景下提供了良好的并发性能。
3.4 安全设计
本系统的安全设计从多个层次进行考虑，构建了纵深防御体系：
（1）传输层安全。系统应部署在HTTPS协议之上，使用TLS加密保护客户端与服务器之间的所有通信，防止中间人攻击和会话劫持。JWT Token和文件内容在传输层获得双重保护。
（2）密码存储安全。用户密码使用pbkdf2_sha256算法进行哈希后存储，该算法采用多轮迭代的密钥派生机制，内置随机盐值，能有效抵御彩虹表攻击和暴力破解。密码明文永不以任何形式存储或记录到日志中。
（3）密钥管理安全。ECC私钥在浏览器端生成后，经用户口令和PBKDF2算法（60万次迭代）派生加密密钥后加密，存储在浏览器localStorage中。加密私钥和加密密钥的明文仅在用户正确输入口令后的会话内存中存在，登出或关闭页面后即从内存中清除。服务端永不接触私钥明文或加密私钥的明文。私钥支持导出为PEM文件备份，导入时需输入口令进行验证。
（4）文件加密安全。每个文件使用独立的、密码学安全的随机AES-256密钥加密（AES-256-GCM），即使某个文件的AES密钥在未来被破解，其他文件的安全性也不会受到影响。GCM模式同时提供机密性和完整性保护，能够检测并拒绝被篡改的密文。
（5）访问控制安全。所有文件操作API均要求有效的JWT Token认证。文件下载需验证用户在file_keys表中有对应的加密密钥记录，或用户为该文件的原始所有者。文件删除、重命名仅限文件所有者执行。分享操作仅限文件所有者发起。这些权限检查在API层面强制实施。
（6）攻击防护。注册和登录接口实施基于IP的滑动窗口速率限制，有效防御暴力破解和账号枚举攻击。文件上传大小限制为50MB，防止磁盘空间耗尽攻击。CORS策略限制允许跨域请求的来源域名，防止CSRF攻击。操作日志记录所有关键行为，支持事后安全审计。

第4章 系统实现
4.1 开发环境与项目结构
本系统的开发环境配置如下：
操作系统：Windows 11；开发语言：Python 3.12+（后端）、JavaScript ES6+（前端）；后端框架：FastAPI 0.110+；ORM框架：SQLAlchemy 2.0+；数据库：SQLite 3（WAL模式）；认证库：python-jose 3.3+（JWT）、passlib 1.7+（密码哈希）；前端技术栈：原生HTML5、CSS3、JavaScript，Web Crypto API；开发工具：PyCharm Community Edition。
项目目录结构遵循FastAPI推荐的项目组织方式。app/目录为应用主目录，包含main.py（入口）、config.py（配置管理）、database.py（数据库引擎与会话）、models.py（ORM数据模型）、schemas.py（Pydantic校验模型）、auth.py（认证路由与JWT）、files.py（文件操作路由）、users.py（用户操作路由）、utils.py（工具函数）、rate_limit.py（速率限制）等模块。static/目录存放前端静态页面文件，uploads/目录存储加密后的文件。
4.2 后端模块实现
后端基于FastAPI框架实现，通过APIRouter将相关API端点分组管理。各路由模块注册到主应用时指定URL前缀。
4.2.1 应用入口与配置
main.py为应用入口。创建FastAPI实例时指定了应用标题"ECC File Sharing System"和版本号。通过add_middleware方法配置CORS中间件，允许跨域请求的来源域名从CORS_ORIGINS环境变量读取，默认值为"http://localhost:8000"。使用app.mount方法将static/目录挂载为/static路径，提供静态文件服务。应用根路径"/"重定向到/static/index.html，直接进入前端界面。启动时调用config.settings.validate()进行配置校验，确保SECRET_KEY已正确设置，然后调用database.Base.metadata.create_all()自动创建数据库表结构。
4.2.2 数据库连接与会话管理
database.py负责数据库引擎创建和会话管理。根据配置中的DATABASE_URL创建SQLAlchemy引擎，对于SQLite数据库设置check_same_thread=False以允许多线程访问。通过SQLAlchemy的event系统监听connect事件，在每次连接建立时执行两条PRAGMA语句：PRAGMA journal_mode=WAL启用预写式日志，允许读写并发；PRAGMA synchronous=NORMAL降同步级别以提升性能。SessionLocal通过sessionmaker创建，autocommit和autoflush均设为False，由应用代码显式控制事务边界。get_db()是一个生成器函数，作为FastAPI依赖注入使用，确保每个请求获取独立的数据库会话并在请求结束后关闭。基类Base使用SQLAlchemy 2.0的DeclarativeBase风格定义。
4.2.3 数据模型定义
models.py定义了4个ORM模型类，与数据库表一一对应：User类映射users表，包含id、username、password_hash、public_key（LargeBinary）、created_at字段，并定义了owned_files、file_keys、logs三个relationship，均配置cascade="all, delete-orphan"实现级联删除。File类映射files表，包含id、filename、owner_id（外键）、encrypted_path、file_size、created_at字段，在owner_id上建立了复合索引。FileKey类映射file_keys表，是文件与用户之间的多对多关联表，包含file_id（外键）、user_id（外键）、encrypted_key（LargeBinary）字段，在file_id和user_id上分别建立索引。Log类映射logs表，记录操作审计信息，包含user_id（外键）、action、target、ip_address、created_at字段，在user_id上建立索引。所有时间字段使用lambda: datetime.now(timezone.utc)作为默认值，使用UTC时区避免时区问题。
4.2.4 认证模块实现
auth.py实现了用户认证和授权的完整逻辑。核心实现包括：passlib的CryptContext配置为使用pbkdf2_sha256算法，提供verify_password和get_password_hash两个密码操作函数。create_access_token函数使用python-jose库的jwt.encode方法，将包含sub（用户名）和exp（过期时间）的payload编码为JWT Token，签名算法为HS256，过期时间默认24小时。get_current_user是核心的认证依赖函数，通过FastAPI的Depends机制注入到各受保护端点中。该函数从请求的Authorization头解析Bearer Token，使用jwt.decode验证Token有效性和签名，提取用户名后从数据库查询对应用户并返回ORM对象。Token无效或用户不存在时抛出401 HTTPException。
注册端点（POST /auth/register）接收username、password和public_key（PEM格式），先通过速率限制检查，然后验证密码长度（≥6位）和用户名唯一性。通过后将公钥PEM编码为UTF-8字节保存，密码通过pbkdf2_sha256哈希后存储，创建User记录并返回用户信息。登录端点（POST /auth/login）使用OAuth2PasswordRequestForm接收表单格式的用户名和密码，通过速率限制后验证密码正确性，签发JWT Token并记录登录日志。修改密码端点（POST /auth/change-password）接收ChangePasswordRequest（old_password和new_password），验证旧密码正确性和新密码长度后更新密码哈希，记录操作日志。
4.2.5 文件操作模块实现
files.py是系统核心业务逻辑的实现模块，包含7个API端点：
文件上传（POST /files/upload）：作为multipart/form-data接收文件、加密密钥和原始文件名。将上传的加密文件以UUID为文件名存储到uploads/目录，边接收数据边累加文件大小，超过50MB时终止上传并删除临时文件。写入完成后创建File记录和FileKey记录，Commit后写入操作日志。
文件列表（GET /files/list）：查询当前用户拥有的所有文件（owner_id = current_user.id），以及通过file_keys表分享给当前用户的文件（user_id = current_user.id）。结果合并去重后，批量查询所有者User信息填充owner_username字段，返回FileInfo列表。
文件下载（GET /files/{id}/download）：先验证用户对该文件的访问权限（是所有者→file_keys表中有对应记录），然后以StreamingResponse流式返回加密文件内容（每次读取64KB分块），加密AES密钥以Base64编码通过X-Encrypted-Key响应头传递，文件名通过X-Filename响应头传递。采用流式响应的设计避免了大文件的Base64编码和JSON序列化问题，支持任意大小文件的下载。下载完成后记录操作日志。
密钥获取（GET /files/{id}/key）、文件分享（POST /files/{id}/share）、文件删除（DELETE /files/{id}）、文件重命名（PUT /files/{id}/rename）、批量删除（POST /files/batch-delete）等端点已在3.2节的功能模块设计中描述，此处不再赘述。
4.2.6 速率限制模块实现
rate_limit.py实现了基于内存的滑动窗口速率限制器。RateLimiter类维护一个字典（IP:端点 → 请求时间戳列表），每次检查时清理窗口外的过期记录。is_allowed方法在窗口内请求数达到上限时返回False，否则记录时间戳并返回True。系统实例化了三个限流器：register_limiter（5次/60秒）、login_limiter（10次/60秒）、change_pwd_limiter（5次/60秒），分别保护注册、登录、修改密码三个敏感端点。
4.3 前端模块实现
前端为单页Web应用（SPA），由单个HTML文件（index.html，约1200行）承载全部页面。页面通过CSS控制多个视图（注册视图、登录视图、文件管理视图）的显示与隐藏，实现无刷新的页面切换。界面采用现代简约风格设计，使用了渐变色的深色标题栏、卡片式布局、响应式设计等元素。
4.3.1 加密函数实现
前端封装了完整的密码学操作函数集：genECC()使用crypto.subtle.generateKey({name: "ECDH", namedCurve: "P-384"})生成ECC密钥对。pubKeyToPem() / privKeyToPem()将CryptoKey对象导出为PEM格式字符串。importPubKey() / importPrivKey()从PEM字符串导入为CryptoKey对象。genAesKey()生成AES-256-GCM密钥。aesEncrypt()使用AES-GCM加密文件内容，将12字节IV与密文拼接返回Blob。aesDecrypt()从Blob头部提取IV后解密。eccEncrypt()实现了完整的ECIES加密流程：生成临时P-384密钥对 → ECDH deriveBits获取384位共享秘密 → HKDF deriveKey派生AES-256-GCM密钥 → AES-GCM加密AES会话密钥 → 返回"临时公钥PEM||IV Base64||密文Base64"。eccDecrypt()为逆向过程。deriveKeyFromPassword()使用PBKDF2（60万次迭代，SHA-256）从用户口令派生加密密钥。encryptPrivateKey()和decryptPrivateKey()使用口令派生的密钥加解密ECC私钥PEM。
4.3.2 注册与登录流程
注册流程：用户输入用户名和密码 → 调用genECC()生成ECC密钥对 → 将私钥导出为PEM格式 → 使用encryptPrivateKey()用口令加密私钥 → 加密私钥存入localStorage（key为sk_<用户名>）→ 公钥随注册请求提交 → 自动调用登录接口获取Token → 进入文件管理界面。登录流程：用户输入用户名和密码 → 调用loginApi()获取JWT Token → 从localStorage读取加密私钥 → 使用decryptPrivateKey()用口令解密 → 将私钥加载到内存（privateKey变量）→ 进入文件管理界面。
4.3.3 文件操作界面
文件管理界面提供了完整的文件操作功能，通过refreshFileList()函数从服务端获取文件列表并渲染为卡片式布局。每个文件卡片显示文件名、文件ID、所有者、上传时间和操作按钮（下载、分享、删除、重命名）。搜索功能在客户端对已加载的文件列表进行文件名过滤，无需额外请求。批量操作通过复选框选中多个文件，支持批量删除和批量分享。
4.4 关键算法实现细节
以下是系统中关键密码学算法的完整实现描述。所有算法均在浏览器端使用Web Crypto API实现，服务端不参与任何密码学运算。
（1）ECC密钥生成。使用ECDH算法和P-384曲线，密钥对既可以用于ECDH密钥协商（deriveKey和deriveBits操作），也可以导出为SPKI（公钥）和PKCS8（私钥）格式。生成的密钥对中，公钥上传至服务端，私钥用口令加密后存储在浏览器本地。
（2）ECIES加密AES密钥。生成临时ECC密钥对（P-384）→ 用临时私钥和接收方公钥执行ECDH deriveBits（384位）→ 将共享秘密导入为HKDF密钥 → 使用HKDF deriveKey派生AES-256-GCM密钥（info参数："file-sharing-ecies"，用于绑定上下文）→ 生成12字节随机IV → 用AES-GCM加密AES会话密钥 → 打包返回临时公钥PEM、IV和密文。该设计确保了前向安全性：即使接收方的长期私钥泄露，临时密钥对也是一次性的，之前的会话密钥不可恢复。
（3）PBKDF2私钥保护。用户口令通过PBKDF2算法派生256位AES-GCM加密密钥。派生参数：随机16字节盐值、60万次SHA-256迭代。60万次迭代在提供足够暴力破解难度的同时，在用户登录时的等待时间控制在可接受范围内（通常1-2秒）。加密后的私钥数据包含：16字节盐值 + 12字节IV + 密文，以Base64编码存储于localStorage。

第5章 系统测试
5.1 测试环境与方法
测试环境：操作系统 Windows 11 Home China 10.0.22000，CPU Intel Core i7-12700H，内存 16GB，Python 3.12.8，浏览器 Google Chrome 120+。
测试方法包括三类：功能测试验证各功能模块是否按预期工作；安全测试验证系统的安全防护机制是否有效；性能测试评估系统在不同条件下的响应时间和资源使用情况。
5.2 功能测试
对系统的全部核心功能设计了测试用例并逐一验证：
用户注册测试：使用合法用户名和6位以上密码注册，预期返回200并创建用户记录。使用已存在用户名注册，预期返回400提示"Username already registered"。使用少于6位的密码注册，预期返回400提示"密码长度不能少于6位"。测试结果全部通过。
用户登录测试：使用正确的用户名和密码登录，预期返回200和JWT Token。使用错误密码登录，预期返回400提示"Incorrect username or password"。使用未注册用户名登录，预期返回400。测试结果全部通过。
文件上传测试：选择1KB文本文件上传，预期后端存储加密文件，数据库中创建File和FileKey记录。尝试上传超过50MB的文件，预期终止上传并返回413错误。测试结果全部通过。
文件下载测试：上传一个文件后立即下载，预期下载的文件内容与原始文件完全一致（经过加解密后）。非文件所有者且未获得分享的用户尝试下载文件，预期返回403 Forbidden。测试结果全部通过。
文件分享测试：用户A上传文件，分享给用户B，用户B登录后应在文件列表中看到该文件并能成功下载。重复分享给同一用户，预期返回400提示已分享。测试结果全部通过。
文件删除与重命名测试：文件所有者删除文件，预期文件从磁盘和数据库中清除。非所有者删除文件，预期返回403。文件重命名为合法新名称，预期成功。重命名为空名称，预期返回400。测试结果全部通过。
批量操作测试：选中3个文件执行批量删除，预期全部删除。选中2个文件批量分享给用户C，预期用户C成功获得2个文件的访问权限。测试结果全部通过。
用户注销测试：通过DELETE /users/me接口注销账户，预期用户及关联的文件、密钥记录、日志全部级联删除，磁盘上加密文件被清除。测试结果通过。
5.3 安全测试
零知识验证：检查服务端uploads/目录下的文件内容，确认为无法识别的二进制密文数据，数据库中encrypted_key字段存储的是ECIES密文，服务端无任何地方存储私钥或明文文件。测试通过。
JWT安全性：使用伪造的JWT Token访问受保护API，预期返回401。使用过期Token访问API，预期返回401。测试结果全部通过。
速率限制测试：在60秒内向注册接口发送6次请求，前5次正常处理（400或200），第6次返回429 Too Many Requests。登录接口在60秒内发送11次请求，前10次正常，第11次返回429。测试结果全部通过。
SQL注入测试：在用户名和密码字段中注入SQL语句（如"admin" OR 1=1 --"），预期SQLAlchemy ORM的参数化查询机制能有效防御，系统正常返回400而非暴露数据。测试通过。
5.4 性能测试
加密性能测试：文件加密时间与文件大小呈线性关系。1KB文件加密耗时约1ms，1MB文件约50ms，10MB文件约400ms，50MB文件约2s。浏览器端的Web Crypto API使用底层系统库，加解密性能良好。ECC密钥生成（P-384）耗时约50-100ms，ECIES加解密耗时约10ms，PBKDF2私钥解密（60万次迭代）耗时约1-2s，均在用户可接受的等待范围内。
API响应时间测试：注册/登录接口响应时间<50ms（不含加密），文件列表接口（100个文件）响应时间<20ms，文件上传接口（10MB文件）传输时间约800ms，文件下载流式响应首字节<10ms。SQLite WAL模式在多用户并发场景下表现稳定，读写操作未出现锁冲突。

结  论
本文设计并实现了一个基于椭圆曲线密码（ECC）的端到端加密文件共享系统。系统采用混合加密架构，文件内容使用AES-256-GCM对称加密保护机密性和完整性，AES密钥通过ECIES方案使用授权用户的ECC公钥进行安全分发。系统实现了以下主要功能：用户注册与登录、浏览器端ECC密钥对生成与管理、文件的加密上传与解密下载、多用户文件安全分享、批量文件操作、操作审计日志和API速率限制等安全防护机制。
本系统的主要特色在于：（1）实现了真正的"零知识"安全模型，所有密码学操作均在用户浏览器端完成，私钥和文件明文永不出现在服务端，有效防止了服务端数据泄露风险；（2）设计了密钥清单机制实现文件一次加密、多用户安全共享，避免了为每个用户重复加密文件的开销；（3）完整的Web应用实现，兼顾了安全性和用户友好性，普通用户无需专业知识即可使用；（4）前端使用Web Crypto API实现高性能的原生密码学运算，保证了与后端安全模型的一致性。
通过功能测试、安全测试和性能测试，验证了系统在功能完备性、安全防护能力和运行效率方面均达到了预期设计目标。系统代码结构清晰，模块耦合度低，具有一定的可扩展性和可维护性。
本系统仍存在一些可以改进的方向：首先，当前系统使用SQLite作为数据库，适合中小规模部署，如需支持大规模用户和高并发访问，可迁移至PostgreSQL等关系型数据库，并结合Redis缓存优化密钥查询性能；其次，可引入零知识证明技术，使用户能够在不泄露文件内容的前提下向第三方证明文件的存在性和完整性；再次，可增加文件版本管理功能，支持加密文件的历史版本回溯；最后，可开发移动客户端应用，利用移动端的生物识别技术（指纹、面部识别）替代口令进行私钥解密，进一步改善用户体验和安全性。
